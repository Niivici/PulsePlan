# PulsePlan 上线前评估与建设文档

> 评估日期：2026-04-15 | 项目版本：1.0.0 | 包名：com.yangyx.pulseplan | 目标平台：HarmonyOS 6.0.0(20)

---

## 一、项目当前状态分析

### 1.1 项目概况

| 维度 | 状态 |
|------|------|
| 项目类型 | HarmonyOS ArkTS 原生应用（Stage 模型） |
| 技术栈 | ArkTS / ArkUI / HarmonyOS SDK 6.0.0(20) |
| 构建工具 | hvigor |
| 包管理 | ohpm（oh-package.json5） |
| 代码规范 | code-linter（@performance/recommended + @typescript-eslint/recommended + 安全规则） |
| 设备支持 | 仅 phone（module.json5 中 deviceTypes 仅含 phone） |
| 当前阶段 | **高保真单页原型**，可演示但不可上线 |

### 1.2 功能完成度评估

| 功能模块 | 原型级 | 生产级 | 缺口说明 |
|----------|--------|--------|----------|
| 五段悬浮导航 | ✅ | ⚠️ | 动画与交互完整，但缺少路由体系支撑 |
| 日/周/月三视图 | ✅ | ❌ | 视图切换可用，但数据全为内存态，无持久化 |
| 周视图拖拽排期 | ✅ | ❌ | 拖拽交互已实现，缺少冲突检测与自动吸附 |
| AI 一键编排 | ✅ | ❌ | 基于关键词匹配的本地模拟，未接入真实 AI 服务 |
| 应急插入与重排 | ✅ | ❌ | 逻辑可用，但无持久化与冲突校验 |
| 执行直达 | ✅ | ❌ | 仅展示文案，未接入系统级应用跳转 |
| 情景模式 | ✅ | ❌ | 仅状态标记，未接入系统免打扰/专注模式 API |
| 成就反馈 | ✅ | ❌ | 徽章与曲线为硬编码，无真实行为数据驱动 |
| 个人中心 | ✅ | ❌ | 静态页面，无账号体系与设置持久化 |
| 手表协同 | ❌ | ❌ | 仅有文案描述，无分布式能力实现 |
| 语音录入 | ❌ | ❌ | 仅有图标占位，无语音识别接入 |

### 1.3 架构现状

- **状态管理**：全部集中在 `PulsePlanHome` 单组件内（约 630 行），使用 `@State` / `@Prop` / `@StorageLink` 管理状态
- **数据层**：`PulsePlanSeed.ets` 提供硬编码种子数据与本地 AI 模拟逻辑，无真实数据源
- **持久化**：仅 `pulseplan_view_mode` 通过 `PersistentStorage` 持久化，其余数据均随页面销毁丢失
- **路由**：单页结构，`main_pages.json` 仅注册 `pages/Index`，无多页面路由
- **组件化**：组件拆分合理（6 个业务组件 + 1 个基础组件），但组件间通过回调链传递数据，缺少全局状态管理

## 三、性能优化要点

### 3.1 渲染性能

| 问题 | 位置 | 优化方案 |
|------|------|----------|
| 24h 时间轴全量渲染 | `TimelineBoard.ets` | 仅渲染可视区域 ± 缓冲区间，虚拟滚动按需加载小时行 |
| `ForEach` 无键优化 | 多处使用 `ForEach` | 确保所有 `ForEach` 的 key 生成函数稳定且唯一，避免全量重绘 |
| 动画过多叠加 | `OverviewSection` / `AchievementSection` | 合并同类动画，减少 `setTimeout` 嵌套，使用 `animateTo` 统一调度 |
| `scheduleItems.map()` 全量复制 | `PulsePlanHome.ets` 拖拽/完成操作 | 使用索引更新替代全量 map 复制，减少 GC 压力 |

### 3.2 内存管理

| 问题 | 优化方案 |
|------|----------|
| 种子数据每次 `aboutToAppear` 重新创建 | 缓存或懒加载种子数据 |
| `chatMessages` 数组无上限 | 设置消息列表最大条数，超出后淘汰旧消息 |
| `setTimeout` / `clearTimeout` 可能泄漏 | 统一在 `aboutToDisappear` 中清理所有定时器引用 |

### 3.3 启动性能

| 问题 | 优化方案 |
|------|----------|
| 启动窗口背景色为白色（`#FFFFFF`） | 改为深色（`#071019`）匹配应用主题，避免白屏闪烁 |
| 首帧加载全量组件 | 拆分懒加载模块，非首屏组件延迟初始化 |

## 五、安全漏洞修复

### 5.1 代码安全

| 编号 | 风险项 | 当前状态 | 修复方案 |
|------|--------|----------|----------|
| S-1 | 代码混淆未启用 | `obfuscation.enable: false` | 上线前设为 `true`，验证混淆后功能正常 |
| S-2 | 日志泄露 | `hilog` 使用 `%{public}s` 输出业务数据 | 生产环境移除或降级日志级别，敏感信息使用 `%{private}s` |
| S-3 | 硬编码用户信息 | `ProfileSection` 中用户名"青锋杨"硬编码 | 从用户服务动态获取 |
| S-4 | 硬编码颜色/密钥 | `PulsePalette` 中所有颜色值硬编码 | 颜色无安全风险，但若后续引入 API Key 需使用安全存储 |
| S-5 | 网络通信安全 | 当前无网络请求 | 接入 AI 服务时必须 HTTPS + 证书校验，禁止 HTTP 明文 |

### 5.2 数据安全

| 编号 | 风险项 | 修复方案 |
|------|--------|----------|
| S-6 | 本地数据未加密 | 用户日程数据持久化时使用 RDB 加密模式 |

| S-8 | 分布式数据传输 | 手表协同场景需使用安全通道传输用户数据 |

### 6.2 发布流程

```
1. 代码冻结 → 创建 release 分支
2. 执行 code-linter 全量扫描 → 0 error
3. 执行单元测试 + UI 测试 → 全部通过
4. 构建 release 包（启用混淆 + 签名）
5. 在真机上执行冒烟测试（覆盖 P0 设备列表）

7. 审核通过后灰度发布（5% → 20% → 50% → 100%）
8. 监控线上指标 48h 无异常后全量发布
```

### 6.3 版本管理规范

- 主版本号：重大功能变更或不兼容更新
- 次版本号：新增功能，向下兼容
- 修订号：Bug 修复与小优化
- 每次发版必须更新 `versionCode`（递增）与 `versionName`

## 七、监控告警机制建立

### 7.1 崩溃监控

| 监控项 | 实现方案 |
|--------|----------|
| JS 崩溃 | 接入 `@kit.PerformanceAnalysisKit` 的 `hilog` + 自建崩溃上报模块 |
| Native 崩溃 | 依赖 HarmonyOS 系统级崩溃采集 |
| ANR 监控 | 主线程超时检测，超过 5s 上报 |

### 7.2 性能监控

| 指标 | 目标值 | 采集方式 |
|------|--------|----------|
| 冷启动时间 | < 2s | `Ability.onCreate` → 首帧渲染完成 |
| 页面切换耗时 | < 300ms | 导航切换时间戳差值 |
| 帧率（FPS） | ≥ 55fps | `@kit.PerformanceAnalysisKit` FPS 采集 |
| 内存占用 | < 200MB | `performance.memory` 监控 |

### 7.3 业务监控

| 指标 | 采集方式 |
|------|----------|
| AI 编排成功率 | `track('ai_plan_generated')` / `track('ai_plan_failed')` |
| 日程创建量 | `track('schedule_created')` |
| 拖拽操作频率 | `track('schedule_dragged')` |
| 完成率趋势 | `getCompletionRate()` 按日聚合上报 |
| 用户留存 | 启动次数 + 活跃天数 |

### 7.4 告警规则

| 规则 | 阈值 | 动作 |
|------|------|------|
| 崩溃率 | > 1% | 立即通知，阻断发布 |
| AI 服务失败率 | > 5% | 降级为本地编排，通知后端 |
| 启动耗时 | > 3s（P95） | 通知性能优化团队 |
| 日活下降 | > 20%（环比） | 通知产品团队排查 |

> 当前 `track()` 函数仅输出 `console.info`，上线前需替换为真实埋点 SDK。
